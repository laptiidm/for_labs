# 📅 System Analysis: Project Time-Slotter

## 1. Огляд проєкту (Executive Summary)
🧩 Ідея проєкту Time-Slotter

Time-Slotter — це веб-застосунок для керування та бронювання часових слотів.
Система дозволяє користувачам переглядати доступний час і бронювати його, а адміністраторам — створювати, редагувати та контролювати ці слоти.

🎯 Основна мета

Надати простий інструмент для:

організації запису клієнтів (наприклад: консультації, послуги)
управління доступністю часу
демонстрації базових принципів розробки (ООП, API, розділення логіки)
**Time-Slotter** — це легка система керування часовими ресурсами (слотами). Вона призначена для автоматизації процесу бронювання часу між клієнтами (Users) та надавачами послуг (Admins). Проєкт демонструє принципи чистої архітектури, розділення ролей та керування станом об'єктів у розподіленому середовищі.

## 2. Бізнес-цілі (Business Goals)
**Автоматизація:** Усунення ручного узгодження часу через месенджери.
**Контроль доступності:** Гарантія того, що один слот не може бути заброньований двічі (Concurrency control).
**Масштабованість:** Можливість використання в різних доменах (консультації, медицина, оренда приміщень).

## 4. Специфікація Use Cases (Functional Requirements)

### [UC-1] Перегляд доступних слотів
* **Актор:** User
* **Pre-conditions:** Система має ініціалізовані слоти в БД.
* **Потік:**
    1. Система фільтрує об'єкти `Slot` за прапорцем `IsBooked == false`.
    2. Дані сортуються за `StartTime`.
* **Post-conditions:** Користувач бачить актуальну сітку доступності.

### [UC-2] Бронювання слоту (Критичний шлях)
* **Актор:** User
* **Бізнес-правило:** Атомарна операція. Якщо два користувачі одночасно тиснуть "Забронювати", доступ отримує той, чий запит прийшов на мілісекунду раніше (First-come, first-served).
* **Потік:**
    1. Валідація ID слоту.
    2. Перевірка статусу `IsBooked`.
    3. Оновлення статусу на `true`.
* **Exception:** Якщо `IsBooked == true`, система повертає `409 Conflict`.

### [UC-3] Адміністрування слотів
* **Актор:** Admin
* **Опис:** Повний CRUD (Create, Read, Update, Delete) життєвого циклу слота. Адмін бачить повну картину, включаючи коментарі та історію бронювань.

## 6. Технічна реалізація (Technical Mapping)

### 🧱 Модель даних (Domain Entity)
```csharp
public class Slot {
    public Guid Id { get; set; }
    public DateTime StartTime { get; set; }
    public DateTime EndTime { get; set; }
    public bool IsBooked { get; set; }
    public string? Comment { get; set; }
}
```

### 🔗 API Design (RESTful)

| Method | Endpoint | Use Case | Description |
| :--- | :--- | :--- | :--- |
| **GET** | `/api/slots/available` | UC-1 | Повертає лише вільні слоти |
| **POST** | `/api/slots/{id}/book` | UC-2 | Змінює стан слоту на "Зайнято" |
| **POST** | `/api/admin/slots` | UC-3 | Створення нового слоту (Admin only) |
| **GET** | `/api/admin/slots` | UC-5 | Повертає всі слоти для звітності |

## 7. Ризики та обробка помилок
1.  **Race Condition:** Вирішується через Optimistic Concurrency в БД (RowVersion).
2.  **Часові зони:** Всі дати зберігаються в **UTC**, конвертація відбувається на боці клієнта.
3.  **Валідація:** `StartTime` не може бути в минулому. `EndTime` має бути пізніше за `StartTime`.
